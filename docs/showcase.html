ThIOClaw — a transparency harness for LLM-powered vulnerability investigation

Security teams are adopting LLM-driven detection & response as opaque vendor features. ThIOClaw exists because that trade-off is unacceptable: detection logic, agent behavior, and reasoning traces belong in source control — not in a black box.

The true intent

ThIOClaw is not a product. It is a workbench — a forkable harness that lets a security team layer an LLM agent on top of deterministic telemetry analysis, while keeping every reasoning step, every tool call, and every verdict auditable, reproducible, and version-controlled.

The core architectural bet: split investigation into two tiers. Tier 1 is deterministic pandas math producing a reproducible verdict. Tier 2 is an LLM agent reasoning on top of the Tier 1 result, gated by Human-In-The-Loop approval before it can propose new queries or update detection logic. The LLM never controls the scoring math — so the same telemetry always yields the same baseline, and you can measure exactly what the LLM contributes on top.

Principle 01

Transparency

Every tool call, evidence lookup, and verdict is logged as structured JSONL and OTel spans.

Principle 02

Repeatability

Tier 1 verdicts are pure pandas math — same input, same verdict, every time.

Principle 03

Configurability

Rules, weights, thresholds, and exploit context live in YAML — no hidden dashboards.

Principle 04

Version Control

Detection logic, prompts, and tools are git-diffable code. Review, approve, roll back.

Principle 05

Measurability

Run the same investigation across models & prompts. Compare Tier 1 baseline against Tier 2 reasoning.

Architecture

The orchestrator dispatches each enabled CVE target into its own data plane module via importlib. The data plane writes a typed tier1.json artifact, then invokes the LLM agent as a subprocess. The agent reads back the deterministic results via its tools, may pause for analyst approval, and finally submits a verdict that is persisted as YAML, Markdown, HTML, and an append-only JSONL log.

The Tier 1 / Tier 2 split — why it matters

This is the single most important design decision in the project. It is what makes the LLM measurable rather than authoritative.

Tier 1 — deterministic data plane

Plain Python & pandas. No LLM. Same input, same verdict, forever.

  • Weighted signal scoring from signals/*.yaml
  • Verdict states: exploited · suspicious · benign · inconclusive
  • Reproducible across runs, models, and operators
  • Works even if the LLM is unavailable

Tier 2 — LLM agent on top

LiteLLM agentic loop — Ollama, Anthropic, OpenAI, Bedrock, etc.

  • Correlates Tier 1 signals against theoretical exploit chain
  • Requests deeper evidence on a per-signal basis
  • Proposes new queries — gated by analyst approval
  • Cannot override Tier 1 math; only annotates and recommends

The bundled example: CVE-2026-31431

The repo ships with one reference implementation — a Linux kernel algif_aead (AF_ALG, socket family 38) local privilege escalation. The signal table below is defined in signals/CVE-2026-31431.yaml and executed deterministically by data_plane/cve_2026_31431.py.

QuerySignalWeightTier
Q1algif_aead kernel module loaded in inventory0.3suspicious
Q2Unprivileged AF_ALG socket open (uid>0)0.5suspicious
Q3UID escalation after AF_ALG open in same PID (primary)1.0exploited
Q4Root shell spawned from non-root parent process0.9exploited
Q5algif_aead module load event in telemetry0.4suspicious
Q6Exploit staging in /tmp, /dev/shm, or memfd0.6suspicious

Verdict logic: exploited if any exploited-tier signal fires AND total weight ≥ 1.0; suspicious if total ≥ 0.5; benign if total = 0; else inconclusive. The sample telemetry in data/sample_events.json exercises the full chain: Q1 module load → Q2 AF_ALG open → Q3 UID escalation → Q4 root shell → Q5/Q6 staging.

HITL — the gate the LLM can’t skip

When the agent decides existing queries are insufficient, it must call propose_query_execution and articulate its rationale and cost. The analyst sees this in the terminal and decides. A second approval is required to persist the new query into the signature file.

[OpenClaw Agent] Proposing Query Execution:
Rationale: The existing Q6 staging query missed /var/tmp paths…

Stack

Control plane

  • LiteLLM — unified tool-calling across providers
  • System prompt + 5 declarative tool schemas
  • Default model: ollama/llama3.1:8b (private, local)
  • Configurable via OPENCLAW_MODEL & OPENCLAW_BASE_URL

Data plane

  • pandas queries over JSON event telemetry
  • Local file source today; S3 manifest scaffolded
  • SQLite-backed inventory (mtime-refresh)
  • One module per CVE — deletable, reviewable, diff-friendly

Observability

  • Structured logs/agent_runs.jsonl
  • Prometheus scrape on :9090/metrics
  • 9 metric instruments (runs, durations, signals, verdicts…)
  • OTel traces — stdout default, OTLP gRPC configurable

Outputs & reports

  • Per-run YAML finding & Markdown/HTML report
  • Append-only findings/findings.jsonl
  • Auto-rebuilt GitHub Pages index after each cycle
  • Pytest suite for ingester, docs builder, signal detection

Run it

python3 -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt

ollama serve && ollama pull llama3.1:8b   # or: export OPENCLAW_MODEL=claude-3-5-sonnet-20241022

python -m harness.orchestrator --raw-telemetry local --once
python -m harness.orchestrator --cve CVE-2026-31431 --once

Approve HITL prompts as they appear in the terminal. Inspect findings/, docs/, and logs/agent_runs.jsonl for the full trace of what the agent did and why.
